# 09_PROMPT_CONSTITUTION.md

## Purpose
This document establishes the Prompt Constitution for JARVIS OS. It defines the immutable system prompting standards and instruction formatting for all LLMs used during development and execution.

## Scope
Applies to system prompts, task instructions, few-shot examples, and dynamic context injections for both internal agents and external IDE coding agents (Cursor, Claude Code, Windsurf).

## Prompt Rules & Formats

### 1. No Conversational Fluff
- **Rule:** Prompts must be direct, command-driven, and technical. 
- **Formatting:** Avoid phrases like "Could you please help me write..." or "I hope you are doing well." Use structured sections: `<objective>`, `<context>`, `<rules>`, `<tasks>`.

### 2. Immutable System Prompt Anchor
- Every system prompt generated by the router must begin with this exact header block:
```
[JARVIS OS SYSTEM PROMPT - ANCHOR]
You are an internal agent of JARVIS OS. 
You must strictly follow 00_PROJECT_CONSTITUTION.md, 08_AI_AGENT_CONSTITUTION.md, and the engineering governance rules under the `.antigravity/` directory.
You are running in role: [INSERT_AGENT_ROLE]. 
Do not answer general questions. Focus strictly on your assigned task.
```

### 3. Drift Prevention Rules
- **Anti-Drift Guidelines:**
  - Enforce schema validations on agent outputs.
  - Limit the input instruction length to prevent model attention dilution.
  - Inject negative prompts to block undesirable behaviors (e.g. "Do not write any code in your response" for the Planner).

## Responsibilities
- **Model Router Module:** Injects the correct prompt layouts and rules before dispatching calls to cloud/local LLM endpoints.
- **Human Developer:** Audits the prompt templates regularly to ensure formatting consistency.

## Dependencies
- Must strictly adhere to the [00_PROJECT_CONSTITUTION.md](file:///e:/jarvis/docs/00_PROJECT_CONSTITUTION.md).

## Interfaces
- Input: JSON prompt configurations loaded from the configuration system.
- Output: Rendered prompt strings sent to LLM APIs.

## Examples
- **Correct Prompt Format:**
```xml
<objective>Decompose lead scrape goal</objective>
<context>docs/03_PRODUCT_REQUIREMENTS.md</context>
<rules>Do not generate code. Follow Rule 11 of Constitution.</rules>
<tasks>
1. Identify scrape target sites.
2. Outline memory save format.
</tasks>
```
- **Incorrect Prompt Format:**
"Hi there, could you help me write a plan to scrape leads and save them to a file? Make sure it is nice and easy to read. Thanks!" (Violates Fluff Rule).

## Failure Cases
- **Prompt Drift:** The LLM starts ignoring instructions after several messages. *Mitigation:* We restart agent memory contexts regularly (see `10_CONTEXT_LOADING_RULES.md`) and inject structural rules in every user prompt iteration.

## Security Considerations
- Prompt templates must never contain API keys, hostnames, or plaintext passwords. All variables must be parsed at runtime from the Secrets Vault.

## Future Extension
- Prompt templates are updated via ADR revisions to prevent silent changes to system instructions.
