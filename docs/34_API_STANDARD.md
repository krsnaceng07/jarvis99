# 34_API_STANDARD.md

## Purpose
This document defines the API Standard for JARVIS OS. It establishes the specifications for REST request/response envelopes, error payload shapes, and real-time WebSocket communication schemas.

## Scope
Applies to all HTTP and WebSocket endpoints created in the FastAPI backend or consumed by the Next.js and Electron interfaces.

## API Standards & Payload Schemas

### 1. REST Response Envelopes
All successful REST API responses must wrap results in a standard JSON envelope:
```json
{
  "success": true,
  "data": {},
  "meta": {
    "timestamp": "2026-06-26T22:30:00Z",
    "request_id": "uuid-string"
  }
}
```

### 2. Error Payload Standard
Error responses (HTTP 4xx/5xx) must follow the standard envelope format:
```json
{
  "success": false,
  "error": {
    "code": "ERROR_CODE_STRING",
    "message": "Human-readable error description.",
    "details": {}
  },
  "meta": {
    "timestamp": "2026-06-26T22:30:00Z",
    "request_id": "uuid-string"
  }
}
```

### 3. WebSocket Real-Time Event Protocol
WebSocket event payloads must use a typed action-payload envelope:
```json
{
  "event": "agent_state_changed",
  "payload": {
    "agent_id": "uuid-string",
    "previous_state": "idle",
    "current_state": "planning"
  },
  "timestamp": "2026-06-26T22:30:00Z"
}
```

## Responsibilities
- **API Developer:** Implements controllers matching these response and error schemas.
- **UI Client Developer:** Standardizes state parsing to handle these envelopes.

## Dependencies
- Must strictly adhere to the [00_PROJECT_CONSTITUTION.md](file:///e:/jarvis/docs/00_PROJECT_CONSTITUTION.md).

## Interfaces
- OpenAPI Schema: Autogenerated by FastAPI and verified during build.

## Examples
- **Correct API Output:** GET `/api/v1/health` returns `200 OK` wrapped in a `"success": true` JSON payload.
- **Incorrect API Output:** GET `/api/v1/health` returns raw string `"OK"`. (Violates REST Envelope rule).

## Failure Cases
- **Schema Drift:** A subagent creates a custom endpoint that returns raw database rows directly. *Mitigation:* The Quality Gates check the autogenerated `openapi.json` file against a validation schema during CI/CD to block deployments if deviations exist.

## Security Considerations
- Error details payloads must never expose database structure details, stack traces, or credentials to the client.

## Future Extension
- Modifications to core envelopes must be documented in matching ADR files.
