# 46_RELEASE_POLICY.md

## Purpose
This document defines the Release Policy for JARVIS OS. It establishes the Semantic Versioning (SemVer) standard, automated changelog generation, and production release tag workflows.

## Scope
Applies to all production deployments, release branch tags, and package distribution files.

## Release Policy & SemVer Standards
1. **Semantic Versioning Standard:** All releases must follow the SemVer specification: `MAJOR.MINOR.PATCH`
   - **MAJOR:** Incompatible API changes or system rewrites (e.g. migrating database engine).
   - **MINOR:** Backwards-compatible new features (e.g. adding new tool).
   - **PATCH:** Backwards-compatible bug fixes and optimizations.
2. **Automated Changelog compilation:** Every release tag must be accompanied by an autogenerated changelog summary detailing:
   - Features implemented (linked to issue IDs).
   - Bugs resolved.
   - Database migrations required.
3. **Release Tagging Workflow:** Releases must be created using Git tags prefixed with `v` (e.g. `v1.0.0`). Tags are cryptographically signed using GPG keys managed by the release manager.

## Responsibilities
- **Release Manager (Human):** Verifies changelogs, signs release tags, and authorizes production deployment.
- **CI/CD Build Runner:** Automates version increments and drafts release packages.

## Dependencies
- Must strictly adhere to the [00_PROJECT_CONSTITUTION.md](file:///e:/jarvis/docs/00_PROJECT_CONSTITUTION.md).

## Interfaces
- Repository API: Git tag endpoints.
- UI: Release dashboard showing current version statuses.

## Examples
- **Correct Release Versioning:** System adds dynamic skill compiling without breaking existing tool signatures -> releases version `v1.2.0`.
- **Incorrect Release Versioning:** System patches a typo in logging files and updates version to `v2.0.0`. (Violates SemVer PATCH rule).

## Failure Cases
- **Bypassed Quality Gates:** A release tag is created but fails compilation in production. *Mitigation:* The deployment runner automatically rolls back production target services to the previous active release version (e.g. from `v1.2.0` back to `v1.1.5`) and logs the failure (see `48_FAILSAFE_AND_ROLLBACK.md`).

## Security Considerations
- Release artifacts (Docker images, Electron binaries) are scanned for security vulnerabilities and signed by the build runner before publication.

## Future Extension
- Modifying release loops or package distribution formats is managed via ADR logs.
