# 98_PHASE_36_SWARM_INTELLIGENCE_SPECIFICATION.md

## Purpose
This document establishes the architecture specification for **Phase 36: Swarm Intelligence & Multi-Agent Consensus**. It defines the protocols, interfaces, schemas, and REST APIs allowing multiple federated nodes and local subagents to propose system-level changes, coordinate votes, and reach cryptographically signed consensus on execution plans, vault access, and node peer membership.

## 1. Architectural Position

The Consensus Subsystem acts as a coordination layer over the Federation layer (Phase 31) and Scale Manager (Phase 35), ensuring that cluster-wide changes are agreed upon by a majority of active federated nodes:

```
     Node A (Proposer)                                   Node B (Voter)
┌───────────────────────────┐                       ┌───────────────────────────┐
│     ConsensusManager      │                       │     ConsensusManager      │
│  (Creates Proposal 123)   │                       │   (Validates & Votes)     │
│            │              │                       │            ▲              │
│            ▼              │  POST /consensus/vote │            │              │
│    FederationManager      ├──────────────────────>│    Federation API Route   │
│ (HMAC signs broadcast)    │                       │   (Authenticates request) │
└───────────────────────────┘                       └───────────────────────────┘
```

## 2. Component Contracts

### 2.1 ConsensusManager

A new service coordinating proposal creation, peer voting, signature validation, consensus tracking, and proposal execution.

```python
class ConsensusManager:
    """Manages multi-agent consensus proposals, voting rounds, and state commitments."""

    def __init__(
        self,
        settings: Any,
        db_manager: Any,
        federation_manager: Any,
        vault_manager: Any,
        event_bus: Any,
    ) -> None:
        """Initialize ConsensusManager with system dependencies."""

    async def create_proposal(
        self, proposal_type: str, payload: Dict[str, Any], proposer_node_id: str
    ) -> Dict[str, Any]:
        """Propose a system mutation (e.g. peer authorization, configuration change).

        Returns:
            The proposal record containing proposal_id, status, and metadata.
        """

    async def cast_vote(
        self, proposal_id: str, voter_node_id: str, vote: bool, signature: str
    ) -> Dict[str, Any]:
        """Record a signed vote from a federated peer.

        Returns:
            The updated proposal status and consensus resolution outcome.
        """

    async def get_proposal_status(self, proposal_id: str) -> Dict[str, Any]:
        """Fetch the current state, voting statistics, and expiration details of a proposal."""
```

## 3. Proposal State Machine

Proposals transition through the following lifecycle based on votes received and expirations:

```text
       CREATED
          │
          ▼
       PENDING
          │
          ├───────────────► APPROVED (Majority >50% reached)
          ├───────────────► REJECTED (Majority opposed or timeout)
          └───────────────► EXPIRED (Vote timeout window exceeded)
```

## 4. REST Endpoint Specifications

All endpoints are mounted under `/api/v1/federation/consensus` and are protected using the P2P request signature validation middleware:

### 4.1 Propose Consensus Action
- **Path**: `POST /api/v1/federation/consensus/propose`
- **Security**: Requires a valid P2P HMAC signature.
- **Request Body**: `{ "type": "config_update", "payload": { "key": "val" }, "proposer_id": "node_a" }`
- **Response**: `{ "status": "PENDING", "proposal_id": "...", "votes": {} }`

### 4.2 Cast Vote
- **Path**: `POST /api/v1/federation/consensus/vote`
- **Security**: Requires a valid P2P HMAC signature.
- **Request Body**: `{ "proposal_id": "...", "voter_id": "node_b", "vote": true, "signature": "..." }`
- **Response**: `{ "status": "APPROVED", "proposal_id": "...", "yes_votes": 2, "no_votes": 0 }`

### 4.3 Query Proposal Status
- **Path**: `GET /api/v1/federation/consensus/proposals/{proposal_id}`
- **Security**: Requires a valid P2P HMAC signature or administrator credentials.
- **Response**: `{ "proposal_id": "...", "status": "PENDING", "yes_votes": 1, "no_votes": 0, "expires_at": "..." }`

## 5. Security & Consistency Invariants
- **Cryptographic Votes**: Each vote must carry an HMAC-SHA256 signature generated by the voter node's shared secret, verified by the proposer using `VaultManager`.
- **Double Voting Prevention**: A peer node is strictly restricted to one vote per proposal ID. Subsequent votes override previous entries but do not duplicate counts.
- **Timeout Rule**: Proposals carry a strict expiration window (default 30 seconds). Votes received after the expiration window are rejected.

---

## 6. Verification and Acceptance Criteria
- **Majority Agreement**: Verify a proposal transitions to `APPROVED` immediately upon receiving yes votes from more than 50% of registered peer nodes.
- **Vote Fraud Rejection**: Verify votes with incorrect signatures or timestamp replay variations are rejected.
- **Strict Expirations**: Verify proposals default to `EXPIRED`/`REJECTED` if vote requirements are not met within the 30-second window.
