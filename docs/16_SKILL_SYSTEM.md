# 16_SKILL_SYSTEM.md

## Purpose
This document defines the Skill System of JARVIS OS. It governs how the system dynamically designs, generates, tests, sandboxes, signs, and registers new capabilities (skills) at runtime.

## Scope
Applies to all dynamically generated plugins, the Skill Registry database, and the sandbox environments where skills run.

## Dynamic Skill Lifecycle
The skill compilation and installation pipeline must strictly follow this security-gated lifecycle:

```
Need New Skill
      ↓
Search Existing Skill
      ↓
Found? → [YES] → Use Skill
      ↓ [NO]
Design Skill Interface
      ↓
Generate Python / JS Code
      ↓
Compile & Test inside Sandbox
      ↓
Run Security Vulnerability Scan
      ↓
Pass Scan & Tests? → [NO] → Refactor / Self-Heal
      ↓ [YES]
Digitally Sign Skill Package
      ↓
Install into Core Registry
      ↓
Register and cache for future reuse
```

### Installation Standards
1. **No Hot-Reloading Production Core:** Dynamic skills are executed inside sandboxed processes or child containers. They cannot modify the core FastAPI codebase.
2. **Signature Verification:** Installed skills must contain a SHA-256 digital signature generated by the Security Agent.
3. **Rollback State:** If an installed skill throws a fatal error, it is immediately deactivated and removed from the active cache.

## Responsibilities
- **Developer Agent:** Designs and generates code for new skills.
- **Security Agent:** Scans code for vulnerabilities (e.g. system calls, SQL injections) and signs approved skills.
- **Resource Manager:** Allocates sandbox containers for testing and running skills.

## Dependencies
- Must strictly adhere to the [00_PROJECT_CONSTITUTION.md](file:///e:/jarvis/docs/00_PROJECT_CONSTITUTION.md).

## Interfaces
- Skill Interface: Defined by the [17_SKILL_SDK_SPEC.md](file:///e:/jarvis/docs/17_SKILL_SDK_SPEC.md).
- Database: Skill records saved in `skills` Postgres table.

## Examples
- **Correct Flow:** System needs to read a custom Excel sheet -> Designer generates `skills/excel_parser/` -> sandbox runs tests -> scanner confirms no shell imports -> signature is attached -> code is loaded and executed.
- **Incorrect Flow:** Developer writes a raw python script containing `os.system("rm -rf")` and loads it directly into the server runtime memory without sandboxing or scanning. (Violates core security rules).

## Failure Cases
- **Unverified Code Exec:** A malformed script bypasses the signature gate. *Mitigation:* The Tool Engine verifies the digital signature of the skill package at every import attempt. If the signature is missing or corrupted, execution is blocked.

## Security Considerations
- Dynamic skill code is considered untrusted. It runs in an isolated network segment, with disk quotas and limited host directory access.

## Future Extension
- Enhancements to the skill lifecycle must be documented in matching ADR files.
