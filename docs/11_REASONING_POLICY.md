# 11_REASONING_POLICY.md

## Purpose
This document defines the Reasoning Policy of JARVIS OS. It governs how agents plan, reflect, utilize tree-of-thought searching, and manage goal stacks and task prioritization queues.

## Scope
Applies to all internal reasoning loop models, Planner configurations, and execution modules in the Brain Core.

## Reasoning Framework & Policies
1. **Goal Stack Management:** High-level goals are pushed onto a Goal Stack. The system recursively pops the top goal, decomposes it, executes subtasks, and updates the stack.
2. **Reflection & Self-Correction:** After a task completes, the agent must run a reflection step comparing output data with the initial task definition.
3. **Execution Limits:** The maximum depth of the goal stack is limited to 10. The maximum number of self-debugging reflection attempts for a single task is limited to 3.
4. **Task Priority Queue:** Subtasks generated by decomposition are assigned priorities (Critical, High, Medium, Low) based on their position in the dependency graph.

## Responsibilities
- **Reasoner Agent:** Evaluates logic correctness and decides when tasks are complete.
- **Planner Agent:** Creates the goal-tree hierarchy and manages the priority queue.

## Dependencies
- Must strictly adhere to the [00_PROJECT_CONSTITUTION.md](file:///e:/jarvis/docs/00_PROJECT_CONSTITUTION.md) (specifically Rule 2, Rule 6, and Rule 11).

## Interfaces
- Input: Goal requests from REST APIs.
- Output: Task status state logs written to Redis and PostgreSQL.

## Examples
- **Correct Flow:** An agent encounters an error writing a file, pops a "debug" goal onto the stack, reflects on the crash log, fixes the file, verifies the fix, pops the debug goal, and resumes.
- **Incorrect Flow:** An agent enters a loop repeatedly trying to write a file with the same code block, running over 50 times. (Violates the max reflection limit).

## Failure Cases
- **Reflection Loop:** The agent keeps patching code and failing. *Mitigation:* When reflection attempts reach 3, the task status is marked as FAILED, the queue stops, and the issue escalates to the human user for approval.

## Security Considerations
- Reflection loops consume large quantities of LLM tokens. The system monitors cumulative token usage (see `65_COST_GOVERNOR.md`) and halts if limits are exceeded.

## Future Extension
- Modifying reasoning parameters or adding search heuristics requires updating this document via an ADR.
