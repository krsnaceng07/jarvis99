# 77_PHASE_15_PERSISTENT_EXECUTION_SPECIFICATION.md

## Purpose
This document establishes the Architecture Specification for **Phase 15: Persistent Execution & Run Management**. It replaces the in-memory session registries of Phase 14 with a production-grade database persistence layer for both agent goal runs and workflow executions, supporting run history queries, lifecycle metrics tracking, and crash-resume hooks on system restarts.

## 1. Architectural Position

The persistence layer bridges the Gateway Layer (`api/`) and the Core Brain (`core/`) by saving state transitions to the database:

```
                  FastAPI HTTP / WebSocket Gateway
                                │
                                ▼
         api/routes/ (agent.py, health.py, workflow.py)
                                │
                                ▼
    core/tools/repository.py  ←  ExecutionRepository (NEW)
    core/memory/models.py     ←  Run & Execution Tables (NEW)
                                │
                                ▼
                 SQLAlchemy Async Connection Pool
```

## 2. Scope & Boundaries

### In Scope
- **Persistent Storage schemas** for Agent Goal Runs, Workflow Executions, and Workflow Step runs.
- **ExecutionRepository** implementing CRUD, version history mapping, and query logic for executions.
- **Resume-after-restart capability:** An initialization startup hook that scans for stale `EXECUTING` sessions (interrupted by a system crash or power-off) and transitions them to a categorized halt state (e.g. `FAILED` with `failure_type = TimeoutFailure` or `PlannerFailure`).
- **REST query endpoints** mapping REST history lookups under `/api/v1/agent/runs` and `/api/v1/workflows/runs`.
- **EventBus integration:** EventBus listeners persist state transitions generated by `ReasoningExecutionEngine` and `WorkflowOrchestrator` directly.
- **Migration logic** updating Phase 14 route handlers to query and persist to database tables instead of the global in-memory `_agent_runs` dict.

### Out of Scope (Non-Goals)
- ❌ User authentication, authorization, RBAC, OAuth, or tenant isolation.
- ❌ API Keys management or route-level rate limiting.
- ❌ Core engine modification (the execution loops in `ReasoningExecutionEngine` and `WorkflowOrchestrator` remain frozen).
- ❌ Multi-user execution workspace allocation.

## 3. Database Schema Specifications

The database models are implemented as SQLAlchemy declarative schemas extending the base model `Base` (imported from `core.memory.models`).

### 3.1 Agent Runs Table (`agent_runs`)
Stores historical and active goal runs execution state.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | `Uuid` | Primary Key | Session run UUID |
| `goal` | `String(4000)` | Not Null | The inbound task goal string |
| `budget` | `Numeric(10, 4)`| Not Null | Upper bound budget limit |
| `state` | `String(50)` | Not Null | `SessionState` string representation |
| `metrics` | `JSONB` | Nullable | Serialized `EngineMetrics` snapshot |
| `failure_type` | `String(100)` | Nullable | `FailureType` categorization |
| `created_at` | `DateTime` | Not Null, Default UTC | Creation timestamp |
| `updated_at` | `DateTime` | Not Null, Default UTC | Last updated timestamp |

### 3.2 Workflow Executions Table (`workflow_executions`)
Tracks compiled workflow runs execution state.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | `Uuid` | Primary Key | Unique execution run UUID |
| `workflow_id` | `Uuid` | Foreign Key, Not Null | Reference to the compiled workflow ID |
| `version` | `Integer` | Not Null | The version index executed |
| `state` | `String(50)` | Not Null | `WorkflowState` string representation |
| `metrics` | `JSONB` | Nullable | Serialized `WorkflowMetrics` snapshot |
| `created_at` | `DateTime` | Not Null, Default UTC | Creation timestamp |
| `updated_at` | `DateTime` | Not Null, Default UTC | Last updated timestamp |

### 3.3 Workflow Step Executions Table (`workflow_step_executions`)
Details wave-level step executions and recovery records.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | `Integer` | PK, Autoincrement | Step run index ID |
| `execution_id` | `Uuid` | FK, Not Null | Reference to the parent workflow run |
| `step_name` | `String(255)` | Not Null | Name identifier of the workflow step |
| `state` | `String(50)` | Not Null | `WorkflowStepState` representation |
| `output` | `JSONB` | Nullable | Output variables returned |
| `error` | `JSONB` | Nullable | Mapped error details if failed |
| `attempts` | `Integer` | Not Null, Default 1 | Number of retry attempts executed |
| `created_at` | `DateTime` | Not Null, Default UTC | Creation timestamp |
| `updated_at` | `DateTime` | Not Null, Default UTC | Last updated timestamp |

### 4.2 Startup Resume Hook (`ResumeManager`)
Integrates with the `lifespan` startup hook of `api/main.py`:
- Scans both `agent_runs` and `workflow_executions` tables for entries stuck in `EXECUTING` or `RUNNING` status.
- Transitions stuck runs to `FAILED` status, logging `failure_type = TimeoutFailure` or `PlannerFailure`.
- Ensures that on a system restart, active states reflect reality and do not hang indefinitely.

## 5. API Gateway Interface Changes

The REST interface remains backward compatible with Phase 14 (`v1`). Endpoints are modified internally to leverage `ExecutionRepository` rather than the in-memory dictionary.

### Modified Endpoints (Internals Updated)
1. `POST /api/v1/agent/run`
   - Persists a new database run record in the `PLANNING` state before spawning background tasks.
2. `GET /api/v1/agent/runs/{run_id}`
   - Resolves status by querying the database repository. Returns 404 if not found.
3. `GET /api/v1/workflows/{workflow_id}`
   - Queries active compilation status records from the repository.

### New Endpoints (History Queries)
1. `GET /api/v1/agent/runs`
   - Returns a paginated list of historical agent runs.
   - Query params: `limit: int = 20`, `offset: int = 0`.
   - Returns: `SuccessEnvelope[List[AgentRunStatusResponse]]`.

## 7. Frozen Architecture Statement

This specification preserves all structural layering invariants. The `api/` layer imports from `core/` to resolve repositories, while `core/` remains strictly agnostic of the API gateway. The schema tables are registered on the shared SQLAlchemy metadata baseline (`Base` class in `core/memory/models.py`) to prevent metadata drift.
